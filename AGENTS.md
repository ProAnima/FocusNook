## Обязательные проверки перед тем, как считать изменение готовым

Это не пожелания — команды ниже must pass чисто (0 errors) перед коммитом:

```
cd apps/desktop
npm run lint      # ESLint: границы слоёв + бюджеты строк, автоматически
npx tsc --noEmit  # типы
npm test          # Vitest
cd src-tauri
cargo clippy --all-targets  # unwrap_used/expect_used/panic/todo — deny, см. Cargo.toml [lints.clippy]
```

`npm run lint` физически не даст импортировать `invoke`/store вне `src/shared/commands.ts` (правило `no-restricted-imports` в `eslint.config.js`) и предупредит про файлы/функции сверх бюджета ниже. `cargo clippy` физически завалит сборку на `unwrap()`/`expect()`/`panic!()`/`todo!()` вне явно помеченного `#[allow(...)]` исключения (bootstrap в `lib.rs::run()`). Если линтер/clippy предлагают исключение — обосновывать явным `// eslint-disable` / `#[allow(...)]` с комментарием, а не молча игнорировать вывод.

## Границы слоёв (обязательно)

- Компоненты React никогда не вызывают `invoke()` или API Tauri-плагинов напрямую — только через `src/shared/commands.ts` (`commandClient`-подобный слой). Смотри существующий `commands.overlay`, `commands.settings`.
- `#[tauri::command]` в Rust — только: валидация входа → вызов use case/доменной функции → typed-результат. Как только появится `crates/planner-core` (раздел 7 архитектурного документа), бизнес-логика переезжает туда, а не растёт в `src-tauri/src`.
- Общий код кладётся в `src/shared/` (`commands`, `theme`, будущие `i18n`, `date-time`); фичи не лезут во внутренности друг друга напрямую.
- UI не хранит и не решает доменное состояние (статусы задач, конфликты синхронизации и т.д.) — это придёт из use cases, когда появится `planner-core`/SQLite.

## SOLID в этой кодовой базе

- **S (Single Responsibility)** — один компонент/модуль = одна причина для изменения. `App.tsx` — только оркестрация shell (табы/настройки/слой), содержимое вкладок выносится в `components/*` (уже так: `DayView`, `EmptyState`, `SettingsPanel`).
- **O (Open/Closed)** — новый sync/AI provider добавляется как новая реализация порта (`SyncProvider` и подобные traits из архитектурного документа), а не веткой `if/else` в существующем коде.
- **L (Liskov)** — любая реализация порта (provider, будущий `AudioCapture` и т.п.) заменяема без изменений в коде, который её вызывает.
- **I (Interface Segregation)** — `commandClient` дробится по возможностям (`overlay`, `settings`, в будущем `planItems`, `reminders`, ...), не собирается в один God-object.
- **D (Dependency Inversion)** — UI и use cases зависят от абстракций (`commandClient`, traits), не от конкретных Tauri/rusqlite/provider реализаций напрямую.

## Бюджеты кода

- Performance-бюджеты из раздела 20 архитектурного документа обязательны к учёту (cold start ≤ 800 ms, idle RAM ≤ 120 MB и т.д.) — новая тяжёлая зависимость или фоновый процесс проверяются на соответствие им, а не добавляются вслепую.
- Файл компонента/модуля — до ~250–300 строк (`max-lines`, ESLint warn); больше почти всегда значит "пора разделить по ответственности".
- Функция/команда — до ~40 строк (`max-lines-per-function`, ESLint warn).
- Новая npm/cargo-зависимость добавляется только если оправдывает вес (см. bundle-бюджеты раздела 20), а не ради одной мелкой функции.

## Тесты обязательны для нового кода

- Frontend: Vitest + React Testing Library уже подключены (`npm test` в `apps/desktop`). Любая новая небезвизуальная логика (обработчики, вычисления, хуки) — минимум unit/component-тест по образцу `src/components/DayView.test.tsx`.
- Rust: `cargo test` — для доменной логики, как только появится `planner-core`. Команды, управляющие реальным окном (`set_overlay_layer` и подобные), пока тестируются вручную через живой запуск — мок-инфраструктуры для `tauri::WebviewWindow` ещё нет; если пропускаешь автотест по этой причине, явно фиксируй это в коммите/PR, а не молчи.
- Ни один коммит не должен снижать существующее покрытие без явного обоснования в описании изменений.

## Прод-уровень — проверять перед тем, как считать код готовым

- Никаких `unwrap()`/`expect()` в Rust вне `main`/`run()` bootstrap — везде `Result` с осмысленной обработкой ошибок.
- Секреты/токены никогда не коммитятся; OAuth — только через OS keychain/Tauri Stronghold (раздел 16 документа), когда появится auth.
- Новые интерактивные элементы — с доступными именами (`aria-label`/`title`) и работают с клавиатуры (раздел 21 документа, a11y).
- Любая новая UI-фича проверяется вручную перед тем как считается готовой: вёрстка/логика — через browser-preview на Vite dev server (`localhost:1420`), нативное поведение окна (transparency, always-on-top, drag) — через реальный запуск `desktop.exe`, потому что preview этого не покажет. Компиляция без ошибок — не то же самое, что "работает".
- Не имитировать законченность: если функциональность частична (например, `DayView` сейчас на статичных данных без записи в SQLite) — явно писать об этом в коде/коммите, а не выдавать за готовую фичу.
